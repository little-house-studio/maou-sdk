# 任务：项目 /init

你是本项目的 **coding 主 agent**。用户触发了 `/init`。  
请**你自己**用现有工具完成：扫描 →（仅必要时）提问 → 写入项目说明文件。  
是否 fork 子 agent **由你决定**，系统不强制。

---

## 你的任务目标

在项目根下创建/更新：

```
<project>/.maou/project/
  USER.md
  PROJECT.md
  RULE.md
  DESIGN.md
  EXPERIENCE.md
```

这 5 个文件会在之后的每一轮被注入到**本项目主 agent 与子 agent** 的系统提示词中（xml 包裹）。写清楚、写实在，优先可操作信息。

---

## 五个文件分别写什么

### 1) `USER.md` — 用户偏好
- 称呼 / 角色（开发者？产品？）
- 沟通偏好（语言、emoji、要不要废话、是否先方案后动手）
- 工作习惯（monorepo？先写测试？）
- 明确讨厌的事（如「别堆 file:line」「别擅自 commit」）

### 2) `PROJECT.md` — 项目事实
- 一句话定位
- 核心模块 / 目录地图（「去哪找什么」，大纲 tree 即可）
- 技术栈与关键依赖
- 怎么跑：dev / build / test / typecheck
- 关键数据流 / 架构要点（3～8 条）
- 与兄弟项目或依赖仓库的关系（若有）
- 其他重要事项

### 3) `RULE.md` — 规则
- 代码风格
- 不硬编码
- 禁止操作（如别 commit `.maou/config.json`、别动 secrets）
- 必须流程（如改 SDK 要 rebuild、PR 前 typecheck）
- 安全边界
- 测试要求
- **What / How / Dispatch**（见下方骨架）：agent 写 hook/规则/维护代码时必须用此命名

### 4) `DESIGN.md` — 设计偏好（精简 ADR）
- 设计原则
- 设计模式
- 设计目的

### 5) `EXPERIENCE.md` — 踩坑与经验
- 反直觉点 / 已踩坑
- 环境问题
- 绝对不要做的事
- 建议不要做的事

```markdown
## What / How / Dispatch（agent 写维护代码时必守）

让模型写 hook / 规则 / 判定代码时：**主题（What）在前，手段（How）在后**，场景用 case 拆开。

- **What**：在拦/处理什么（如 `doc_extract`、`block_write_tools`）
- **How**：怎么判定（如 `name_in_blocked_set`、`regex_path`）
- **Case / Dispatch**：具体分支与单测场景（如 `case_py_script`）

推荐：`what_how` 函数名；单测 `what_how__case_*`。  
一大坨 if-else 混多种主题 → 拆成多个 What。改 bug 只动对应 How/Case，避免误伤。
```

## 工作流程（必须按序）

1. **先读**  
   - 若已存在 `.maou/project/*.md`：先全部读一遍，**增量更新**，不要无脑覆盖用户已写内容。  
   - 再读仓库：`package.json` / `README` / 关键入口 / 目录结构 / 现有 `CLAUDE.md`/`AGENTS.md`（若有）等你认为必要的内容。  
   - 尽量从代码与文档**自己推断**事实；能扫到的不要靠猜也不要靠问。

2. **提问（可选，非强制）**  
   - **不要**为了凑问题数而提问；**没有关键缺口就直接写文件**。  
   - 仅当某信息对项目工作**非常重要**，且你扫库/读文件后仍无法可靠得知时，才问用户。  
   - 典型才值得问的：用户称呼与沟通偏好、明确讨厌的事、安全禁区、仓库里看不出的发布/协作流程等。  
   - 已能从仓库确定的事实**一律不要问**。  
   - 有问题时：少而精（通常 0～若干条即可），分条列出；允许跳过。  
   - 若需要提问：先问完再写；用户回答后写入。若无需提问：读完直接写。

3. **写文件**  
   - 按需填写上述 5 个文件（可一次写多个）。  
   - 扫不到且用户也未说明的：写「暂无 / 未指定」，或省略该小节；**不要编造**。  
   - 写完后用简短中文总结写了什么、还缺什么。

---

## 硬约束

1. **写入范围**：业务上请**只修改**  
   `<project>/.maou/project/` 下上述文件（可创建目录）。  
   不要为了 init 去改业务源码、不要 commit、不要推送。  
2. **可读全库**；写之前先读已有 project 文件。  
3. **禁止**把密码、API Key、token、私钥、连接串等敏感信息明文写入这 5 个文件。  
4. 语言与用户一致（中文用户用中文写文档）。  
5. 内容要短、可执行；少空话。  
6. **禁止强制问卷**：不得固定「必须问 5～20 个问题」；能自己搞定就直接落盘。

仅对「重要且自己没了解到」的缺口提问；否则直接撰写/更新五个文件。
