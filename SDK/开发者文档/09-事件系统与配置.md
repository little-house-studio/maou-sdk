# 事件系统、配置与安全

---

## 事件系统

### LLM 事件流

```
send → first_token → output_start → delta* → output_end → receive
               └─ thinking_start → thinking* → thinking_end ─┘
```

| 事件 | 触发时机 | 携带数据 |
|------|----------|----------|
| `send` | 用户消息已加入队列 | `ChatMessage` |
| `first_token` | 首字输出 | `{ content: string }` |
| `thinking_start` | 模型开始推理 | — |
| `thinking` | 每次推理增量 | `{ thinking, thinkingContent }` |
| `thinking_end` | 推理结束 | `{ thinkingContent }` |
| `output_start` | 正式输出开始 | — |
| `delta` | 每个文本增量 | `{ delta, content }` |
| `output_end` | 输出完成 | — |
| `receive` | 完整响应 | `ChatMessage`（含 toolCalls、usage） |
| `tool_call` | 检测到工具调用 | 流式：`{ toolName, toolComplete }`，非流式：`{ toolCall }` |
| `error` | 错误 | `{ error: string }` |
| `abort` | 用户中断 | — |
| `parse_error` | JSON 解析异常 | `{ error: string }` |

### ChatDelta 流式字段提取

当 LLM 流式输出结构化 JSON 时，`StreamJsonAccumulator` 实时解析字段：

| delta.type | 携带数据 | 时机 |
|------------|----------|------|
| `field_streaming` | `fieldName` + `fieldContent` + `fieldDelta` | 字段正在流式输出 |
| `field_complete` | `fieldName` + `fieldValue` | 字段完整（引号闭合） |
| `tool_call` | `toolName` | 检测到工具调用关键词 |

## 循环检测

`ModelCaller` 内置 `LoopDetector`，检测 LLM 是否陷入重复输出。

- **原理**：检查输出 buffer 末尾是否存在重复的模式
- **最小模式长度**：8 字符
- **阈值**：默认 10 次重复（可通过 `loopThreshold` 配置）
- **触发**：检测到死循环 → 自动停止当前轮次 → 回报 `loop_detected` 事件

## 安全提示

1. **API Key 管理** — 使用环境变量（`OPENAI_API_KEY`、`DEEPSEEK_API_KEY` 等），不要在代码中硬编码
2. **工具权限** — 通过 `allowedModes` 限制工具在 plan/execute 模式下的可用性。设置 `null` 表示全部可用
3. **沙箱** — 通过 `ToolContext.sandboxRoot` 限制文件系统访问范围，`sandboxMode` 控制安全级别
4. **终端工具** — 默认需要 `sandboxMode` 配置，可设置黑白名单。敏感操作（如 `edit`、`bash`）需审批
5. **并发控制** — `ToolExecutor` 默认最大 10 并发，每工具默认 30s 超时
6. **敏感信息脱敏** — 日志输出会自动脱敏 API Key（`_headers` 中排除 `Authorization`）
7. **429 限流处理** — `LLMClient` 自动处理 429 响应：读取 `Retry-After` 头，指数退避最多 8 次重试
